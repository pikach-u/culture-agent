# CLAUDE.md

## 1부. 코딩 4원칙 (Karpathy 기반)

### 1. 코드 짜기 전에 생각하기

**가정하지 말 것. 혼란 숨기지 말 것. 트레이드오프 드러낼 것.**

- 가정을 *명시*하기. 불확실하면 묻기.
- 여러 해석이 있으면 _침묵하지 말고_ 둘 다 제시. 혼자 골라서 가지 말 것.
- 더 단순한 접근이 있으면 그걸 말하기. 필요하면 푸시백.
- 이해 안 되는 부분이 있으면 멈추기. 무엇이 불명확한지 _이름 붙이고_ 묻기.

### 2. 단순함 우선

**문제를 푸는 _최소_ 코드. 추측성 코드 금지.**

- 요청한 것 _이상으로_ 기능 추가 금지.
- 한 번 쓰는 코드에 추상화 만들지 않기.
- 요청 안 한 "유연성"이나 "설정 가능성" 만들지 않기.
- 일어날 수 없는 시나리오에 에러 처리 만들지 않기.
- 200줄로 짠 게 50줄로 가능하면 다시 쓰기.

판단 기준: "시니어 개발자가 보면 과하다고 할까?" 그렇다면 단순화.

### 3. 외과적 변경

**필요한 것만 건드리기. 본인이 만든 것만 정리.**

기존 코드 수정 시:

- 옆에 있는 코드/주석/포맷을 "개선"하지 않기.
- 망가지지 않은 걸 리팩토링하지 않기.
- 본인이라면 다르게 할 스타일이라도 _기존 스타일에 맞추기_.
- 무관한 죽은 코드 발견하면 _언급만_ 하고 삭제는 하지 않기.

본인 변경이 만든 고아 처리:

- 본인 변경으로 안 쓰이게 된 import/변수/함수는 제거.
- 기존부터 죽어있던 코드는 별도 요청 없으면 안 건드림.

판단 기준: 변경된 모든 줄이 사용자 요청에 *직접 추적*되어야 함.

### 4. 목표 기반 실행

**검증 가능한 성공 기준 정의. 그 안에서 루프.**

명령형 → 검증 가능한 목표로 변환:

- "검증 추가" → "잘못된 입력 케이스 명시 + 실제로 거절되는지 확인"
- "버그 고쳐" → "재현 케이스 명시 + 그 케이스가 동작하게 만들기"
- "X 리팩토링" → "리팩토링 전후 동작 동일성 확인 기준 명시"

> **본 프로젝트 주석**: 자동 테스트 인프라 미도입 (학습 단계). 위 패턴은 _수동 DoD_ 형식으로 적용 — 사용자가 PROGRESS의 DoD 체크박스로 직접 검증.

다단계 작업은 짧은 plan으로 시작:

```
1. [단계] → 검증: [확인]
2. [단계] → 검증: [확인]
```

## 2부. 학습 프로젝트 추가 원칙

1. **계획 → 승인 → 실행**: 코드 짜기 전에 무엇을/왜 할지 _글로_ 설명. 사용자 OK 후 구현. 큰 변경은 단계별 쪼개서 매 단계 승인.

2. **"왜?"를 항상 설명**: 라이브러리 선택, 함수 구조, 비동기 여부 등 결정의 이유 명시. 학습 가치 보존이 본 프로젝트 동기.

3. **사실 확인은 도구로**: 패키지 버전, API 가격, 파일 내용, SDK 사용법 등은 _추측 금지_. 검색/실측한 결과만 단언. 모르면 모른다고 말하기.

4. **노션 백업 친화적 포맷**: 정리된 내용은 마크다운으로 깔끔하게 (헤딩, 표, 코드블록). 평소 대화는 자연스럽게 — 매 답변마다 과한 마크다운 X.

5. **단계 범위 엄수**: 사용자가 명시한 Stage 범위만 작업. 다음 Stage 코드 미리 만들지 않음.

6. **진단보다 단순한 길 우선**: 원인이 단순할 가능성이 높을 땐 복잡한 진단 명령 안 짜고 "혹시 X 아냐?" 한 줄로 확인.
   _Stage 4 교훈_: 모델 갈아끼우기 무한 루프에 빠지면 멈추고 다른 접근(예: 컨텍스트 주입) 고려.

7. **클로드코드 명세 패턴**: 코드 작업은 클로드코드한테, 결정/리뷰는 사용자+클로드. 명세는 _새 결정사항만_ 명시 (컨텍스트로 추론 가능한 건 빼고). 작업 원칙(자율성 제한, 자동 실행 금지, 모호하면 묻기)은 매번 명시.

## 3부. 본 프로젝트 누적 함정

### 환경 (Windows 10 + PowerShell)

- 모든 진입점 `.py`에 `sys.stdout.reconfigure(encoding="utf-8")` 필요. 안 하면 `cp949` 에러.
- venv 활성화는 새 터미널마다 필요. 프롬프트 앞 `(venv)` 매번 확인. `ModuleNotFoundError: telegram` 류 에러 = venv 미활성 1순위.
- `.env` 편집은 UTF-8 에디터 (VS Code) 사용. 메모장 ANSI 저장 금지.
- `python -c` 인라인 한글 깨짐 방지: `$env:PYTHONIOENCODING='utf-8'` 선설정.

### Ollama 통합 (Stage 4 학습)

- **모델 카드의 능력 ≠ 런타임 통합 상태**. 모델이 "function calling 지원"이라 적혀도 ollama chat template 통합이 안 되어 있을 수 있음.
- ollama 라이브러리 페이지의 **"Tools" 태그가 통합 완성 신호**. 태그 없으면 능력 있어도 raw text로 누출.
- **모델 패밀리 같다고 통합도 같지 않음** (qwen3 vs qwen3.5).
- "안 됨" 출처 분리: HF GGUF 직접 import vs ollama 공식 라이브러리 vs 특정 사이즈는 모두 다른 경로. 일반화 금지.
- ollama 데몬과 Python SDK는 **별개 버전** (데몬 0.23.x / SDK 0.6.x).
- ollama 데몬은 두 프로세스 (`ollama app.exe` 트레이 + `ollama.exe` 서버). 둘 다 종료해야 완전히 끔.
- **OOM 발생 시**: 다른 프로그램 일부 종료로 0.2~1 GiB 확보. 영구 발생 시 `NUM_CTX 8192 → 4096` 검토 (KV 캐시 절반).

### 시스템 프롬프트 작성 (Stage 5/7 학습)

- 작은 모델은 instruction을 영역별로 잘 _compartmentalize 못 함_. 한 영역의 brevity/persona 지시가 다른 영역으로 누출.
- 결정적인 두 가지: **어휘 선택** + **스코프 명시**. 예: "특화된" → "특히 강합니다" (페르소나 좁히지 않기), "(추천 요청일 때만 적용)" 헤더 (스코프 분리).
- *부정 명령*은 작은 모델에 효과 있음 (Stage 7): "절대 'X 했어요' 같은 거짓 응답 하지 마세요"는 환각 방지에 작동.

### Google API SCOPES (Stage 7 학습)

- **`calendar.events`는 events.list/insert만 커버**. `freebusy.query`는 `calendar.readonly` 별도 필요. 둘 다 명시해야 둘 다 동작.
- SCOPES 변경 시 기존 `token.json` 무효 → 삭제 후 `/connect` 재실행 필수.
- 일반화: API 권한은 "엔드포인트별 매핑" 직접 확인. "도메인 같으니 같은 scope일 것" 추측 금지.

## 4부. PROGRESS.md 갱신 규칙

PROGRESS.md는 _작업 일지_. 비대화 방지를 위한 규칙.

### 새 Stage 항목 형식 (10줄 이내)

```
### Stage N (이름, 날짜)
- **의도**: (1줄)
- **구현**: (3~5줄, 핵심 결정만)
- **검증**: (1~2줄, DoD 통과 여부)
- **의도된 한계**: (1줄, 후순위 처리 항목)
```

### 어디에 무엇을 적는가

- **CLAUDE.md (이 파일)**: 일반 원칙, 다른 Stage에서도 통용되는 함정/교훈.
- **PROGRESS.md**: Stage별 _프로젝트 특정_ 결정과 검증 결과.

### 슬림화 타이밍

- 직전 Stage 1개만 "마지막 작업" 섹션에 자세히. 그 이전은 결정 기록(7번)으로 압축 이동.
- DoD 체크박스는 직전 Stage 1~2개만 보존. 통과 후 다음 Stage 진입 시 단계별 체크리스트(5번)로 압축.
- 결정 기록(7번)은 Stage별 1~5줄 핵심만. 세부 설명은 일반 원칙이면 CLAUDE.md로 이동.

### Stage 끝날 때 확인 질문

- 새로 발견한 *일반 원칙*이 있나? → CLAUDE.md 2부 또는 3부로.
- 직전 Stage 항목이 10줄 넘게 부풀었나? → 핵심만 남기고 압축.
- 트러블슈팅 표에 새 row 추가했나? → 일반화 가능하면 CLAUDE.md 3부로 이동.
